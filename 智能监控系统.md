# 智能视频监控与行为分析系统（基于 YOLOv8 & rtsp2web）

## 需求分析与项目计划书

### 一、 项目概述

本项目旨在开发一套集实时视频流转发、边缘/云端 AI 智能行为分析、以及历史视频录制回放于一体的智能监控系统。系统通过接入前端摄像头的 RTSP 原始流，利用 YOLOv8 进行实时的目标检测与行为分析，并将带有 AI 识别框的画面进行**“双轨制”**分发：

1. **实时轨**：通过流媒体服务器与 `rtsp2web` 服务，实现 Web 端的超低延迟（接近实时）直播。
2. **回放轨**：将 AI 分析画面按固定时间切片保存为 MP4 文件，实现长达 30 天的录像缓存与 Web 端常规点播。

------

### 二、 需求分析 (Requirement Analysis)

#### 2.1 功能性需求

- **AI 智能分析**：系统需能够拉取多路 RTSP 监控流，使用 YOLOv8 进行目标检测和特定行为（如跌倒、打架、越界等）的实时推理，并将分析结果（识别框、标签、警报文字）实时绘制在视频帧上。
- **低延迟实时直播**：前端 Web 页面需能够低延迟播放带有 AI 分析画面的视频流。
- **历史录像切片与存储**：系统需将带有 AI 画面的视频按固定周期（建议 1 小时/个）自动存为 MP4 格式文件至服务器磁盘。
- **历史录像回放**：前端用户可根据日期、时间段选择历史监控视频进行点播观看，支持快进、拖拽进度条。
- **存储生命周期管理**：系统需自动清理生成时间超过 30 天的历史视频文件，释放磁盘空间。

#### 2.2 非功能性需求

- **算力保障**：AI 分析模块需部署在配备足够显存的 GPU 服务器上，以保障 YOLOv8 逐帧推理的实时性。
- **高并发与稳定性**：`rtsp2web` 结合 WebSocket 需能稳定支撑多路并发观看。
- **跨平台兼容性**：前端播放不依赖任何浏览器插件，纯 HTML5/JavaScript 实现。

------

### 三、 系统架构与技术栈设计

本项目采用**后端转码 + AI 算力服务 + WebSocket 传输 + 前端渲染 + Nginx 静态代理**的混合架构。

#### 3.1 核心技术栈

| **模块**         | **技术选型**                       | **职责说明**                                                 |
| ---------------- | ---------------------------------- | ------------------------------------------------------------ |
| **AI 推理服务**  | Python, OpenCV, YOLOv8             | 拉流、抽帧、模型推理、结果绘制、本地录制及重新推流。         |
| **流媒体服务**   | MediaMTX 或 SRS                    | 接收 Python 脚本推上来的新 RTSP 流，作为中转。               |
| **实时转发引擎** | Node.js, `rtsp2web`, FFmpeg, ws    | 将 RTSP 流转为 mpeg1 或 flv 格式，通过 WebSocket 推送至前端。 |
| **前端实时播放** | Vue/React, `jsmpeg.js` 或 `flv.js` | 接收 WebSocket 二进制流，使用 `<canvas>` 或原生 `<video>` MSE 渲染画面。 |
| **录像点播服务** | Nginx , HTML5 `<video>`            | 提供 MP4 静态文件的 HTTP 访问，前端直连播放。                |
| **任务调度**     | Linux Cron / Python Schedule       | 自动执行 30 天过期视频的删除任务。                           |

#### 3.2 核心业务流向 (双轨制)

1. **数据源**：摄像头原始 RTSP 视频流。
2. **AI 处理**：Python 脚本拉取原流 $\rightarrow$ YOLOv8 推理 $\rightarrow$ 叠加图层框。
3. **分流 A（看当前）**：Python (FFmpeg/OpenCV VideoWriter) 推送新 RTSP 流 $\rightarrow$ MediaMTX $\rightarrow$ `rtsp2web` Node 服务 $\rightarrow$ WebSocket $\rightarrow$ 浏览器 `jsmpeg`。
4. **分流 B（看历史）**：Python 实时写帧落盘 $\rightarrow$ 按小时切分成 MP4 文件 $\rightarrow$ Nginx 代理目录 $\rightarrow$ 浏览器 HTTP 点播。

### 四、 项目实施计划 (Project Plan)

本项目预计分为四个主要阶段（Sprint），总周期预估视团队规模而定，此处按标准的敏捷开发周期进行规划。

#### 阶段一：AI 引擎与基础流媒体搭建 (基础设施准备)

- **目标**：跑通原始视频源 $\rightarrow$ AI 识别 $\rightarrow$ 输出处理后画面的核心链路。
- **任务清单**：
  1. 搭建 GPU 运行环境，部署 YOLOv8 及相关跟踪算法（如 DeepSORT）。
  2. 编写 Python 脚本，通过 OpenCV 接入测试 RTSP 流。
  3. 完成打架、跌倒等行为分析算法的串联，并将 Bbox (边界框) 和文字绘制在内存帧中。
  4. 部署流媒体服务器（MediaMTX/SRS），测试 Python 脚本将处理后的画面实时推流至该服务器。

#### 阶段二：实时视频流 Web 端展现 (低延迟攻坚)

- **目标**：实现带有 AI 分析画面的视频流在浏览器中秒级延迟播放。
- **任务清单**：
  1. 部署 `rtsp2web` Node.js 服务环境，配置 `fluent-ffmpeg` 底层依赖。
  2. 将阶段一推流到流媒体服务器的 RTSP 地址配置给 `rtsp2web`，启动转码与 WebSocket 服务。
  3. 前端开发：引入 `jsmpeg.js` (或 `flv.js`)，连接 WebSocket，在 `<canvas>` 中渲染实时画面。
  4. 调优：测试延迟，优化 FFmpeg 转码参数（如降低比特率、调整帧率）以确保内网/公网的流畅度。

#### 阶段三：历史录像落盘与回放系统开发 (双轨制闭环)

- **目标**：完成视频切片存储、前端历史回放及过期文件清理逻辑。
- **任务清单**：
  1. 改造 Python 脚本：增加 `cv2.VideoWriter` 逻辑，将带 AI 画面的帧同步写入本地磁盘，并在代码层面实现每满 1 小时自动生成新文件（命名规范：`camId_YYYY-MM-DD_HH00.mp4`）。
  2. 部署 Nginx 静态文件服务器，将录像文件夹映射为 HTTP 可访问目录。
  3. 编写服务器定时脚本（Cron job），每天凌晨 2:00 扫描录像目录，执行 `rm` 删除 30 天前的文件。
  4. 前端开发：增加日期、时间选择组件。选中后，拼接对应的 MP4 URL，使用原生 `<video>` 标签进行点播。

#### 阶段四：集成测试、压测与交付 (系统固化)

- **目标**：确保系统在多路并发条件下的稳定性和健壮性。
- **任务清单**：
  1. **显存与 CPU 压测**：测试单台 GPU 服务器最多能支撑几路 YOLOv8 实时推理，确定服务器资源边界。
  2. **磁盘 IO 测试**：确保多路视频同时落盘时的 I/O 性能不会导致 AI 推理进程卡顿或丢帧。
  3. **网络压测**：多客户端同时连接 WebSocket 时的网络带宽占用测试。
  4. 编写部署文档（Docker 容器化部署建议）和用户操作手册。

### 五、 风险评估与应对策略

1. **GPU 算力瓶颈**
   - *风险*：多路视频同时进行 YOLOv8 实时推理时，GPU 显存溢出或帧率大幅下降。
   - *应对*：采用**跳帧检测策略**（例如每 3 帧检测一次，中间帧采用光流法或目标跟踪算法）；模型量化加速（使用 TensorRT 将模型转为 `.engine` 格式大幅提升推理速度）。
2. **磁盘 I/O 与存储爆炸**
   - *风险*：高清视频持续落盘导致磁盘读写压力过大，或 30 天累计文件过大。
   - *应对*：落盘视频采用 H.264 硬件编码（如 NVIDIA NVENC）减小 CPU 压力并降低文件体积；预先计算 30 天所需磁盘容量并配备足够的 NAS 或企业级监控硬盘。
3. **进程假死崩溃**
   - *风险*：Node.js (`rtsp2web`) 或 Python 推理脚本因异常崩溃中断监控。
   - *应对*：全面引入进程守护工具（如 `PM2` 守护 Node 服务，`Supervisor` 或 Docker 重启策略守护 Python 服务）。
